# 20 ВОПРОСОВ НА СЕНЬОРНОСТЬ

**39.** PEP 669 ввёл `sys.monitoring` с per-code-object событиями. Объясни механику: как CPython подменяет байткод на `INSTRUMENTED_*` варианты и что такое `tstate->tracing`? Почему `sys.monitoring` может дать near-zero overhead там, где `settrace` даёт 40x?

**40.** До PEP 667 (3.13) `frame.f_locals` возвращал **снимок** dict, а `PyFrame_FastToLocalsWithError` копировал fast-locals в него. Из-за этого pdb-команда `x = 5` не работала. Что именно вернул PEP 667 и какой C API сломался (`PyFrame_LocalsToFast`)?

**41.** `sys.settrace` не вызывает `'line'` события в C-функциях и не срабатывает внутри `list.sort(key=f)` до Python 3.? — где реально проходит граница между трассируемым и нетрассируемым кодом? Почему `sys.setprofile` не даёт `'line'` вообще?

**42.** `faulthandler` использует только async-signal-safe вызовы и пишет напрямую в fd через `write(2)`, минуя буферы Python. Почему нельзя использовать обычный `traceback` в обработчике SIGSEGV? Что случится с `PyGILState_Ensure` в этом контексте?

**43.** Процесс на проде висит на 100% CPU. `py-spy dump --pid` показывает стек без остановки процесса, читая память через `process_vm_readv`. Как py-spy находит `PyThreadState` в чужой памяти и почему это ломается при смене минорной версии CPython?

**44.** `PYTHONMALLOC=debug` включает `_PyMem_DebugRawAlloc` с байтами-стражами `0xFD`/`0xCD`/`0xDB`. Что означает каждый паттерн и как по дампу отличить use-after-free от buffer overflow? Какой ценой в RAM и скорости?

**45.** Traceback пойманного исключения удерживает все кадры → цикл ссылок `frame → exc → tb → frame`. Python 3 добавил неявный `del e` в конце `except E as e:`. Почему этого недостаточно, если ты сохранил `e` в переменную выше по стеку, и как это выглядит в долгоживущем воркере?

**46.** `breakpoint()` разрешает `sys.breakpointhook` **при каждом вызове**, а не при импорте. Какой атаке это открывает дверь и почему `PYTHONBREAKPOINT` — вектор произвольного импорта? Что бы ты сделал в security-hardened рантайме?

**47.** В Go есть `dlv`, в Java — JDWP-протокол с attach к живой JVM по сети. У CPython нет штатного wire-протокола отладки. Почему? Что делает `debugpy`/`pydevd` вместо этого и какова цена в overhead?

**48.** `pydevd` (движок PyCharm/VSCode) использует frame evaluation API из PEP 523 (`_PyInterpreterState_SetEvalFrameFunc`) вместо `settrace`. Как это даёт conditional breakpoints почти бесплатно? Что произошло с PEP 523 в 3.11+ со специализацией байткода?

**49.** Ты ставишь breakpoint внутри `__del__`. Отладчик виснет или ведёт себя необъяснимо. Почему? Как связаны GC, отложенные finalizer'ы и реентерантность trace-функции?

**50.** 500M-строчный ETL, падает на ~400M строке раз в сутки в проде, воспроизвести локально нельзя. Твоя архитектура отладки: `faulthandler`, sampling profiler, structured logging с correlation id, core dump + gdb, чекпоинты? Что ставишь первым и какой overhead-бюджет?

**51.** `sys.settrace(f)` внутри самой trace-функции игнорируется — CPython выставляет флаг реентерантности. Как тогда работают отладчики, которые сами исполняют Python-код в обработчике (evaluate expression в watch-окне)? Что происходит с событиями внутри `eval` под отладчиком?

**52.** `pdb` использует `bdb.Bdb` с `botframe`/`stopframe`/`returnframe`. Объясни, как реализована команда `next` — почему она сравнивает `frame` с `stopframe`, а не считает строки? Где эта логика ломается на генераторах?

**53.** Условный breakpoint с `cond` вызывает `eval(cond, frame.f_globals, frame.f_locals)` на **каждом** попадании в строку. Замерь мысленно: цикл 10M итераций, breakpoint с условием `i == 9_999_999`. Во сколько раз медленнее? Как сделать это дёшево через `sys.monitoring` + DISABLE?

**54.** Ты вызываешь `faulthandler.dump_traceback_later(60)` в C-extension, который держит GIL и делает `sleep(120)` в native-коде. Дамп напечатается? Объясни через watchdog-поток `faulthandler` и то, что он не требует GIL.

**55.** До 3.11 traceback указывал только строку. PEP 657 добавил колоночные позиции (`a.b.c().d` → подчёркивание точного вызова). Где хранятся позиции, сколько байт стоит и что делает `-X no_debug_ranges` / `PYTHONNODEBUGRANGES`?

**56.** Post-mortem по core dump: `gdb python core` + `py-bt` из `python-gdb.py`. Как gdb-скрипт восстанавливает Python-стек из C-стека? Почему на релизной сборке без frame pointer'ов и с inlined `_PyEval_EvalFrameDefault` это часто не работает?

**57.** Heisenbug: race в многопоточном коде исчезает под pdb. Объясни через `sys.setswitchinterval(0.005)` и через overhead trace-функции — почему отладчик меняет тайминг переключений? Как ловить такое без изменения тайминга?

**58.** `tracemalloc.start(25)` + `snapshot.compare_to()` находит утечку по строке аллокации. Какой overhead по памяти и CPU при глубине 25? Чего `tracemalloc` принципиально не видит (аллокации в C через `malloc`, `mmap`, буферы numpy)?
