# 20 ВОПРОСОВ НА СЕНЬОРНОСТЬ

**39.** `set_add_entry` в `setobject.c` пробует `LINEAR_PROBES = 9` соседних слотов перед `perturb`-скачком. Обоснуй число 9 через размер cache line и `sizeof(setentry)`. Почему в `dict` этот трюк убрали?

**40.** `set` растёт при `used > mask * 3 / 5`, `dict` — при `used * 3 >= usable * 2`. Оба ≈0.6, но `set` при resize берёт `used * 5` для маленьких и `used * 3` для больших (>50000). Какую патологию хеш-множеств лечит именно множитель 5?

**41.** Хеш `frozenset` строится XOR-ом скрамбленных хешей элементов плюс финальный mix с `len`. Почему нельзя просто XOR? Приведи семейство множеств, которые коллидировали бы при наивном XOR, и оцени DoS-потенциал.

**42.** У `frozenset` хеш кешируется в поле `hash`, инициализированном в `-1`. Как это взаимодействует с `hash()`, возвращающим `-1` для реального значения, и во что это выливается по числу пересчётов для `frozenset` из 1M элементов?

**43.** `d[{1, 2}]` → `TypeError`, но `{1, 2} in {frozenset({1, 2})}` → `True`. Найди место в CPython, которое делает это несимметричным, и объясни, почему такой хак не сделали для `dict`.

**44.** PEP 218 добавил `set` в 2.3 через модуль `sets`, встроенный тип — в 2.4, `frozenset` — тогда же. Что было не так с `sets.Set`/`sets.ImmutableSet` и почему `BaseSet.__hash__` в чистом Python оказался тупиком?

**45.** Литерал `{1, 2, 3}` в `x in {1, 2, 3}` сворачивается в `frozenset`-константу, а `x in [1, 2, 3]` — в `tuple`. Почему список не превратили в `frozenset`, хотя семантика `in` та же? Что мешает при `x` — нехешируемом?

**46.** `set` хранит хеш прямо в `setentry`, `dict` (combined) — тоже. Но при resize `set` не может переиспользовать чужую таблицу, а `dict` умеет split-table для `__dict__` инстансов. Объясни, почему key-sharing принципиально не переносится на множества.

**47.** Java `HashSet` — это обёртка над `HashMap` с dummy-значением `PRESENT`. CPython держит отдельный `setobject.c` в ~2500 строк. Назови три вещи, которые Python выигрывает от отдельной реализации, и цену в поддержке.

**48.** Множество из 500M 64-битных ID, RAM 16GB. Считай: `set` из int — сколько байт на элемент с учётом `PyLongObject` и заполнения таблицы? Какая структура вместо (`pyroaring`, Bloom, sorted array + bisect) и какой запрос это решает/ломает?

**49.** `set` не отдаёт память при удалении элементов — только при resize, а resize вниз происходит при следующей вставке. Напиши воспроизведение утечки-по-дизайну в долгоживущем сервисе и назови, чем это отличается от dict.

**50.** `s1 & s2` итерируется по меньшему множеству, `s1.intersection_update(s2)` — нет (или наоборот?). Проверь по исходникам и объясни, почему на 1M vs 10 элементов разница в 5 порядков и в какую сторону.

**51.** `PYTHONHASHSEED` рандомизирует хеши `str`/`bytes` с 3.3 (PEP 456, SipHash). `int` не рандомизирован — `hash(n) == n` для малых. Атака: злоумышленник контролирует int-ключи, попадающие в `set`. Реализуема ли hash-flooding и что спасает?

**52.** `set.__eq__` — O(min(n, m)), `set.__le__` — тоже. Но `sorted(list_of_sets)` компилируется и не падает. Объясни, почему частичный порядок + Timsort даёт молча неверный результат, и приведи входные данные, где `sorted` вернёт неотсортированное.

**53.** `set` subclass с переопределённым `__hash__`/`__eq__` кладут в `frozenset`. При каком условии элемент «пропадёт» из множества, оставшись в таблице? Как это связано с mutation-after-insert и почему CPython не защищается?

**54.** `dict.keys()` реализует set-протокол, `dict.items()` — тоже, но `items()` ломается на нехешируемых values. `dict.values()` set-операций не имеет вообще. Обоснуй каждое из трёх решений и скажи, где `items() & items()` даст `TypeError` в проде.

**55.** `set.pop()` не рандомен: он сканирует таблицу с `finger`-подобной позиции. Найди, есть ли в CPython `finger` для set (как в `dict.popitem`), и оцени сложность цикла `while s: s.pop()` для множества, где все элементы в конце таблицы.

**56.** `frozenset()` — синглтон, `frozenset(x)`, где `x` уже `frozenset` — возвращает тот же объект. Но `frozenset(x)` для subclass — нет. Найди условие в `make_new_set` и объясни, какой инвариант это защищает.

**57.** `collections.abc.Set._hash()` реализует ровно алгоритм `frozenset_hash` на чистом Python — включая магические константы вроде `1927868237`. Зачем ABC обязали совпадать побитово с C-версией и что сломается при расхождении?

**58.** `set` в PyPy — это `dict` с `None`-значениями плюс strategy-оптимизация (`IntegerSetStrategy`, `UnicodeSetStrategy`). CPython от такого отказался. Разбери trade-off: где PyPy выигрывает на порядок, где проигрывает и почему CPython не берёт strategies.
